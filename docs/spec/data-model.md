# Data Model

All state is stored as a single JSON object under the `localStorage` key `tyov_character`.

---

## Top-level shape

```ts
{
  name:         string,           // Character name, editable in header
  promptNumber: number | '',      // Current prompt number (positive integer or empty)
  createdAt:    string | null,    // ISO 8601, set on first name entry; null until then

  memories:     Memory[],
  skills:       Skill[],
  resources:    Resource[],
  characters:   Character[],
  marks:        Mark[],

  archive: {
    memories:   Memory[],
    skills:     Skill[],
    resources:  Resource[],
    characters: Character[],
    marks:      Mark[],
  }
}
```

---

## Entity definitions

### Memory

```ts
{
  id:          string,        // uid() — base-36 timestamp + random suffix
  title:       string,        // Opening phrase or title of the memory
  experiences: Experience[]   // 0–3 entries
}
```

**Constraints:**
- Max 5 in `state.memories` at any time.
- If a 6th would be added, Resolution Mode fires — user must archive one before the new Memory is created.
- Archived by moving the whole object (with Experiences intact) to `state.archive.memories`.

### Experience

```ts
{
  id:   string,   // uid()
  text: string    // Freeform description of what happened
}
```

**Constraints:**
- Max 3 per Memory.
- Adding a 4th is blocked with an inline "This memory is full." message.
- Removing an Experience does not archive it — it is deleted in place (Experiences have no independent archive).

### Skill

```ts
{
  id:      string,
  text:    string,
  crossed: boolean   // true = lost (strikethrough); still visible in play view
}
```

**Constraints:** None on count.  
**Removal:** Archiving moves the object to `state.archive.skills`. Strikethrough is a non-destructive "lost" marker.

### Resource

```ts
{
  id:      string,
  text:    string,
  crossed: boolean
}
```

Identical structure and rules to Skill.

### Character

```ts
{
  id:     string,
  name:   string,
  desc:   string,    // Short descriptor or note
  mortal: boolean,   // true = Mortal, false = Immortal
  alive:  boolean    // true = Alive, false = Dead
}
```

**Constraints:** None on count.  
**Removal:** Archiving moves the object to `state.archive.characters`. Dead characters are shown with reduced opacity and struck-through name in the play view but remain active until explicitly archived.

### Mark

```ts
{
  id:   string,
  text: string
}
```

**Constraints:** None on count.  
**Removal:** Archiving moves the object to `state.archive.marks`.

---

## IDs

All entity IDs are generated by:

```js
function uid() {
  return Date.now().toString(36) + Math.random().toString(36).slice(2);
}
```

IDs are not guaranteed to be globally unique across saves or devices, but collisions within a single session are negligible.

---

## Archive rules

| Entity | Archive destination | Content preserved |
|---|---|---|
| Memory | `archive.memories` | Full object including all Experiences |
| Skill | `archive.skills` | `id`, `text`, `crossed` state at time of archiving |
| Resource | `archive.resources` | Same as Skill |
| Character | `archive.characters` | Full object including `mortal`, `alive`, `desc` |
| Mark | `archive.marks` | `id`, `text` |

The archive is append-only and read-only in the UI. There is no restore action. The past is permanent.

---

## Persistence

- `save()` — `JSON.stringify(state)` → `localStorage.setItem('tyov_character', ...)`
- `load()` — `JSON.parse(localStorage.getItem('tyov_character'))` → `state`; falls back to `blankState()` on error or absence
- Writes are synchronous and happen on every mutation. There is no debounce, no dirty flag, and no manual save action.
